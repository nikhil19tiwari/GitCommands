1. GIT CONCEPTUAL MODEL: THE FOUR AREAS
================================================================================

To understand how to move files forward and backward, you must understand the
four architectural areas of Git.

A. The Working Directory (or Working Tree):
   DEFINITION: This is the directory on your file system where you see the
   actual files of your project. This is where you are currently editing,
   creating, or deleting files.
   STATUS: Files here are either "Untracked" (newly created) or "Modified"
   (tracked files with unsaved changes).

B. The Staging Area (or Index):
   DEFINITION: A single file, generally located within your .git directory, that
   stores information about what will go into your next commit. It is a concept
   of preparing what the snapshot will look like.
   STATUS: Files here are "Staged."

C. The Local Repository (HEAD):
   DEFINITION: This is where Git permanently stores the metadata and object
   database for your project. When you commit, you are writing a new immutable
   object to this area. It resides within the .git directory.
   STATUS: The HEAD pointer typically points to the latest commit in your
   current branch.

D. The Remote Repository:
   DEFINITION: A version of your project hosted on the internet or network (e.g.,
   GitHub, GitLab).
   STATUS: This is where you push your local changes so others can see them, or
   pull changes others have made.

2. BASIC SETUP AND REPOSITORY INITIALIZATION
================================================================================

COMMAND: git config
   DEFINITION: Sets configuration options for your Git installation on a global
   or repository level. Crucial for identifying commits.
   Example (Global Setup):
     $ git config --global user.name "John Doe"
     $ git config --global user.email "johndoe@example.com"

COMMAND: git init
   DEFINITION: Creates a new, blank Git repository by initializing a .git
   directory in the current folder.
   Example:
     $ mkdir my_project
     $ cd my_project
     $ git init

COMMAND: git clone <url>
   DEFINITION: Downloads an existing repository from a remote location to your
   local machine, automatically setting up a remote tracking branch.
   Example:
     $ git clone https://github.com/user/repo.git

3. BASIC WORKFLOW: TAKING A SNAPSHOT
================================================================================

COMMAND: git status
   DEFINITION: Shows the current state of the Working Directory and the Staging
   Area. It tells you which files are modified, untracked, or staged.
   Example:
     $ git status

COMMAND: git add <file>
   DEFINITION: Adds a file from the Working Directory to the Staging Area.
   This is "Staging" the changes.
   Example (Stage a specific file):
     $ git add index.html
   Example (Stage all modified/deleted/new files in current directory):
     $ git add .

COMMAND: git diff
   DEFINITION: Shows the line-by-line differences between files.
   - Running 'git diff' alone shows changes in the Working Directory that have
     NOT been staged.
   - Running 'git diff --staged' (or --cached) shows the difference between the
     Staging Area and the last commit (what will be committed).

COMMAND: git commit -m "message"
   DEFINITION: Takes a snapshot of the changes currently in the Staging Area and
   writes them permanently to the Local Repository. A message is required.
   Example:
     $ git commit -m "Fix login button typo"

COMMAND: git commit -am "message"
   DEFINITION: A shortcut that adds (git add) all tracked files that have been
   modified and commits them in one step. It does NOT add new "untracked" files.

4. THE "UNDO" BUTTONS: NAVIGATING BACKWARD
================================================================================

This is the core of moving files backwards through the four areas.

SCENARIO 1: Discarding changes in the Working Directory (Undo modifications).

   COMMAND: git restore <file>
   DEFINITION: (Modern Git >= 2.23) Discards changes made to a file in the
   working directory, ***"reverting it back to the state of HEAD || the Staging Area"***.
   WARNING: This cannot be undone.
   Example (Undo uncommitted changes in a file):
     $ git restore index.html
   Alternative (Pre-Git 2.23 syntax):
     $ git checkout -- index.html

SCENARIO 2: Unstaging a file (Staging Area -> Working Directory).

   COMMAND: git restore --staged <file>
   DEFINITION: Removes the specified file from the Staging Area but keeps the
   actual file content in the Working Directory. This "unstages" the changes.
   Example (I added a file, now I want to unstage it):
     $ git add secret.txt      (Moved to staging)
     $ git restore --staged secret.txt  (Moved back to modified in working dir)
   Alternative (Pre-Git 2.23 syntax):
     $ git reset HEAD secret.txt

SCENARIO 3: Undoing commits locally (moving Local Repo -> Staging/Working).

   COMMAND: git reset <options> <commit-hash> (or HEAD~)
   DEFINITION: Resets the current branch head to a specified state, potentially
   altering the index and working directory.

   Option --soft:
     Moves the HEAD back to the specified commit. The changes remain STAGED in
     the Staging Area. Use this if you want to redo the commit message.
     Example:
       $ git reset --soft HEAD~1   (Undoes last commit, changes are staged)

   Option --mixed (Default):
     Moves the HEAD back and resets the Staging Area. The changes remain in the
     WORKING DIRECTORY but are UNSTAGED.
     Example:
       $ git reset HEAD~1   (or git reset --mixed HEAD~1)

   Option --hard:
     Moves the HEAD back, resets the Staging Area, AND overwrites the Working
     Directory. WARNING: Any uncommitted changes are lost forever.
     Example:
       $ git reset --hard HEAD~1

SCENARIO 4: Safely undoing a published (Pushed) commit.

   COMMAND: git revert <commit-hash>
   DEFINITION: Instead of rewriting history (like reset), revert creates a NEW
   commit that introduces the exact opposite changes of the specified commit.
   Use this for commits already pushed to a shared remote.
   Example:
     $ git revert a1b2c3d4   (Creates a new commit undoing a1b2c3d4)

5. BRANCHING AND MERGING
================================================================================

COMMAND: git branch
   DEFINITION: Lists all branches in the current repository.

COMMAND: git branch <new-branch-name>
   DEFINITION: Creates a new branch pointer pointing to the current HEAD commit,
   but does not switch to it.
   Example:
     $ git branch feature/user-profile

COMMAND: git checkout <branch-name> (or modern: git switch <branch-name>)
   DEFINITION: Switches the working directory to the specified branch, updating
   the files to match that branch's latest snapshot.
   Example:
     $ git switch feature/user-profile

COMMAND: git merge <branch-name>
   DEFINITION: Integrates the changes from the specified <branch-name> into the
   CURRENT branch.
   - Fast-forward Merge: If the current branch has not diverged, Git simply
     moves the current branch pointer forward.
   - Three-way Merge: If both branches have diverged, Git creates a new "Merge
     Commit" that integrates both histories. May cause Merge Conflicts.

COMMAND: git branch -d <branch-name>
   DEFINITION: Deletes a branch. Use -D to force delete if the branch hasn't
   been merged yet.

6. STASHING: PAUSING YOUR WORK
================================================================================

CONCEPT: You have changes in your working directory that are not ready to be
committed, but you must switch branches immediately (e.g., to fix a bug).
Committing half-baked code is bad practice.

COMMAND: git stash push (or just git stash)
   DEFINITION: Takes your modified tracked files and staged changes, removes
   them from the working directory/staging area, and saves them onto an internal
   stack within .git.
   Example:
     $ git stash

COMMAND: git stash save "message"
   DEFINITION: Similar to above but allows you to add a description.
   Example:
     $ git stash save "WIP: login logic incomplete"

COMMAND: git stash list
   DEFINITION: Lists all currently stored stashes.
   Example:
     stash@{0}: WIP on main: 5a123f1 Initial commit

COMMAND: git stash apply <stash-id>
   DEFINITION: Re-applies the changes from a stash (defaults to stash@{0}) but
   DOES NOT remove the stash from the stack.

COMMAND: git stash pop
   DEFINITION: Re-applies the changes from the latest stash (stash@{0}) AND
   removes it from the stash stack. This is the most common command.

COMMAND: git stash drop <stash-id>
   DEFINITION: Manually removes a specific stash from the stack without applying it.

COMMAND: git stash clear
   DEFINITION: Deletes all stored stashes.

7. INSPECTING HISTORY (DEEP DIVE INTO GIT LOG)
================================================================================

COMMAND: git log
   DEFINITION: Displays the commit history of the repository, showing the hash,
   author, date, and commit message.

Essential Flags for git log:

   --oneline
     DEFINITION: Displays each commit as a single line (short hash + message).
     Excellent for quick overview.
     Example:
       $ git log --oneline
       5a123f1 Fixed critical bug
       a1b2c3d Refactored UI

   --graph
     DEFINITION: Draws an ASCII graph representing branching and merging history.
     Often used with --oneline.
     Example:
       $ git log --graph --oneline --all
       * 5a123f1 Merge PR #12
       |\
       | * a1b2c3d Updated docs
       |/
       * 8e9f0a1 Base commit

   --all
     DEFINITION: Shows commits from ALL branches, not just the current one.

   -p (Patch)
     DEFINITION: Shows the log entry followed by the 'diff' (actual file changes)
     introduced by that commit.
     Example:
       $ git log -p -2   (Shows the diff for the last 2 commits only)

   --stat
     DEFINITION: Shows statistics for files modified (insertions +, deletions -).

   --author="name"
     DEFINITION: Filters logs to only show commits by a specific author.

   --grep="pattern"
     DEFINITION: Filters logs to show commits whose message matches the pattern.

COMMAND: git blame <file>
   DEFINITION: Shows line-by-line which commit modified that specific line, and
   who made that change. Useful for auditing historical changes in complex files.

8. THE REMOTE WORKFLOW AND COLLABORATIVE COMMANDS
================================================================================

COMMAND: git remote add <name> <url>
   DEFINITION: Associates a name (conventionally 'origin') with a remote URL.
   Example:
     $ git remote add origin https://github.com/user/project.git

COMMAND: git remote -v
   DEFINITION: Lists the configured remotes and their corresponding fetch/push URLs.

COMMAND: git fetch <remote>
   DEFINITION: Downloads all new branches, tags, and commits from the remote
   repository to your local machine. It DOES NOT merge or affect your working
   directory. This is "Remote Repo -> Local objects DB."

   Moving Remote -> Staging Area:
     The user asked how to move from "remote to staging." Fetch brings the data
     locally. To align staging exactly with remote:
       $ git fetch origin          (Pulls objects)
       $ git reset origin/main     (Resets staging/HEAD to origin/main)

COMMAND: git pull <remote> <branch>
   DEFINITION: A combined command that runs 'git fetch' immediately followed by
   'git merge' (or git rebase) to integrate remote changes into your current
   local working branch.
   Example:
     $ git pull origin main

COMMAND: git push <remote> <branch>
   DEFINITION: Uploads the commits in your local branch to the specified remote
   repository and updates the remote-tracking branch.
   Example:
     $ git push origin feature/profile

9. ALTERING HISTORY (AMENDING AND PUSHING CHANGED DATA)
================================================================================

This addresses changing commit messages and data from staging and push.

SCENARIO 1: Changing the last commit message (Locally, not pushed yet).

   COMMAND: git commit --amend -m "new message"
   DEFINITION: Updates the very last commit (HEAD) with the current Staging Area
   content and replaces the old commit message. Creates a new commit object.

   Workflow example:
     $ git commit -m "Fix bugg"           (Commit with typo)
     $ git commit --amend -m "Fix bug"    (Fix message locally)

   Changing data within that last commit:
     $ git commit -m "Fix bug"            (Forgot a file)
     $ git add missing_file.txt
     $ git commit --amend --no-edit       (Updates commit, keeps message)

SCENARIO 2: Changing the last commit message AFTER it has been pushed.

   DEFINITION: You cannot "change" a pushed commit because commits are immutable.
   You must amend locally (creating a new commit with a different hash) and then
   FORCE push this new history to the remote.

   WARNING: Only do this on private branches you control exclusively. Never
   force push to shared branches (like main), as it overwrites colleagues' work.

   Workflow example:
     $ git commit -m "Typo message"
     $ git push origin feature-branch
     (Realize the error)
     $ git commit --amend -m "Correct message"   (Local fix, new hash)
     $ git push origin feature-branch --force    (Overwrite remote)

COMMAND: git rebase -i HEAD~n (Interactive Rebase)
   DEFINITION: Allows you to alter many commits in the past 'n' commits,
   including rewriting messages, squashing multiple commits into one, reordering,
   or deleting commits.

10. PLUMBING: POST-MORTEM OF THE .GIT FOLDER
================================================================================

The .git folder is the engine room of Git. If you delete it, the folder is no
longer a repository. It contains two vital things: the configuration file and
the Object Database.

Key Components of .git:

   /config (File)
     DEFINITION: Project-specific configuration settings, including remote URLs,
     branch tracking info, and local settings (override global ~/.gitconfig).

   /HEAD (File)
     DEFINITION: A plain text pointer telling Git what you are currently on.
     Usually points to the ref of the current branch.
     Example content: "ref: refs/heads/main"

   /index (File)
     DEFINITION: A binary file representing the Staging Area. It caches
     information about which file versions are staged for the next commit.

   /logs/ (Directory)
     DEFINITION: Stores the reflog (reference logs), tracking every time HEAD
     moved or branches changed (crucial for recovering lost work via git reflog).

   /refs/ (Directory)
     DEFINITION: Stores references (pointers) to commits.
     - /refs/heads/: Local branch pointers (e.g., a file 'main' contains the
       latest commit hash for that branch).
     - /refs/remotes/: Remote-tracking branch pointers (e.g., origin/main).
     - /refs/tags/: Pointers for tags.

   /objects/ (Directory)
     DEFINITION: The Object Database. Stores ALL content using SHA-1 hashes.
     Objects are immutable (cannot be changed once written).

THE FOUR TYPES OF GIT OBJECTS (THE CORE DATA STRUCTURES):

   Blobs (Binary Large Objects):
     DEFINITION: Stores only the actual content of a file (the data). Does not
     store filename, directory structure, or permissions. A specific file
     version gets one Blob hash.

   Trees:
     DEFINITION: Acts like a Unix directory. Maps filenames and permissions to
     Blobs and other Trees. A commit points to one root Tree.

   Commits:
     DEFINITION: Binds the snapshot (Tree) to metadata. Contains:
       - Reference to the root Tree (snapshot hash)
       - Author and Committer name/date
       - Commit message
       - References to PARENT commit(s) (this creates history)

   Tags:
     DEFINITION: An annotated tag object (like a commit) pointing to a specific
     commit hash, with its own metadata. Lightweight tags are just refs in
     /refs/tags/ (not full objects).

11. COLLABORATION PLATFORM CONCEPTS (FORK, PR, ISSUE)
================================================================================

These are NOT core Git commands but features provided by platforms like GitHub,
GitLab, and Bitbucket. They facilitate collaboration using core Git.

CONCEPT: Issue (Bug/Feature Request/Task)
   DEFINITION: A tracking mechanism hosted on the platform for project management.
   Users use Issues to report bugs, request new features, or assign tasks.
   Issues can be linked to pull requests.

CONCEPT: Fork
   DEFINITION: A complete, personal copy of someone else's repository, hosted
   under your own account on the platform. It is a severe form of cloning
   (remote to remote).
   PURPOSE: You do not have permission to push changes to the original
   repository. You Fork it, make changes to your copy, and then propose those
   changes to the original owner.

CONCEPT: Pull Request (PR) or Merge Request (MR)
   DEFINITION: The mechanism for proposing changes. When you finish work on a
   branch in your Forked repository, you submit a Pull Request to the original
   (upstream) repository.

   Process:
     1. Fork the repo (Platform action).
     2. Clone your Fork: $ git clone <your-fork-url>
     3. Create feature branch: $ git switch -c feat/update-docs
     4. Commit changes.
     5. Push branch to your Fork: $ git push origin feat/update-docs
     6. On the platform, click "Create Pull Request."
     7. The original project owner reviews your diff, leaves comments, and
        decides whether to accept (merge) your changes into their project.
